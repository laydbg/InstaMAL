Domain characterization: This domain covers managed service provider (MSP) client environments — small-to-medium enterprise networks where a third-party provider administers the infrastructure under a trusted relationship. Defining structural features include a centralized domain controller for authentication, a mix of workstations and servers under domain management, and a privileged MSP administrative identity with universal access across all assets. This domain is particularly interesting from a security perspective because the MSP trust relationship introduces a single high-privilege identity whose compromise yields simultaneous administrative access to the entire environment, creating a uniquely high-value lateral movement shortcut that bypasses normal privilege escalation paths.

Requirements:

1. **Requirement:** Every instance must contain a domain controller asset connected to an internal network and associated with a credential database that stores hashed credentials for all domain accounts, including at least one domain administrator account.
   **Rubric:** For each instance, verify that exactly one asset is typed as a domain controller; that it has a network-membership association to an internal network; and that it has a credential-storage association to a database asset containing at least one domain-admin-level account credential.

2. **Requirement:** Every instance must contain an MSP administrative account asset with elevated-privilege access associations to every machine asset (workstations, servers, and domain controller) in that instance — no machine asset may be unreachable via this account.
   **Rubric:** For each instance, identify the MSP admin account asset; enumerate all machine-typed assets; confirm that a privileged-access association exists between the MSP account and each machine asset. Flag any instance where a machine asset lacks this association.

3. **Requirement:** At least one workstation in every instance must hold credentials for a local user in an in-memory (more accessible) form, in addition to the hashed form in the local account database, representing a credential exposure vector distinct from the database store.
   **Rubric:** For each instance, identify workstation assets; verify that at least one has both a hashed-credential-storage association to a local account database and a separate in-memory-credential association to a local user credential asset.

4. **Requirement:** Instances must vary in the count of workstations and servers. Across the collection, the number of workstations per instance should range across at least three distinct values (e.g., 1–4), and the number of servers per instance should range across at least two distinct values, so that agents do not overfit to a fixed network size.
   **Rubric:** Collect the workstation count and server count for every instance. Confirm that workstation counts take at least three distinct values across the collection and server counts take at least two distinct values. Instances with only a single workstation and single server should not constitute more than a small minority of the collection.

5. **Requirement:** Instances must vary in the number and placement of external network exposure points. Some instances should have only the internal network with a single outbound connection, while others should have additional assets (e.g., a DMZ segment, an externally reachable server, or a VPN gateway) that create additional entry points from the external network.
   **Rubric:** For each instance, count the number of assets that have a network-membership or connectivity association to the external network node. Verify that across the collection this count varies — at least some instances have exactly one such asset and at least some have two or more, with at least one instance containing a distinct intermediate network segment (e.g., DMZ) between the external network and the internal network.

6. **Requirement:** Instances must vary in where domain administrator credentials are accessible beyond the domain controller. In some instances the domain admin credential should be cached or stored only on the domain controller; in others it should additionally appear as a cached or stored credential on one or more workstations or servers, reflecting real-world credential sprawl.
   **Rubric:** For each instance, identify the domain admin credential asset and enumerate all credential-storage or credential-cache associations pointing to it. Across the collection, verify that at least one-third of instances have the domain admin credential accessible only from the domain controller, and at least one-third have it accessible from at least one additional machine, creating variation in the lateral movement path length to that credential.

7. **Requirement:** Instances must include variation in local user privilege levels across machines. In some instances all local users should be standard (non-admin) accounts; in others at least one local user on a workstation or server should hold local administrator privilege, representing a misconfiguration or intentional delegation that shortens privilege escalation paths.
   **Rubric:** For each instance, inspect the privilege-level attribute of all local user account assets. Confirm that across the collection, both configurations are represented: instances where every local user is standard-privilege-only, and instances where at least one local user account carries local-admin privilege. Neither configuration should account for fewer than roughly 20% of the collection.

8. **Requirement:** Every instance must include at least one application asset hosted on a server, with variation across instances in whether those applications have inbound connectivity associations from the external network (i.e., are internet-facing). This captures the attack surface difference between purely internal services and externally exposed ones.
   **Rubric:** For each instance, verify that at least one server asset has an application-hosting association to an application asset. Across the collection, confirm that some instances have no application with an external-network-reachable association, and some have at least one application reachable from the external network node, with the split being neither all-one-way nor trivially uniform.

9. **Requirement:** The MSP admin credential must itself be stored in varying locations across instances. In some instances it should reside only in the domain controller's credential database; in others it should additionally be stored on a dedicated management host or jump server asset, reflecting different MSP operational models and producing different attacker paths to that credential.
   **Rubric:** For each instance, find the MSP admin credential asset and enumerate all assets with a credential-storage association to it. Verify across the collection that at least two distinct storage configurations appear: (a) MSP credential stored only on the domain controller, and (b) MSP credential stored on at least one additional dedicated management or jump-host asset. Configuration (b) should appear in a meaningful fraction of instances (roughly one-quarter or more).
