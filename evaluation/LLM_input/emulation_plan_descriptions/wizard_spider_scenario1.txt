The environment is a small corporate Windows domain containing two user workstations and a domain controller, all connected on a shared internal network. Both workstations have outbound internet access. Domain authentication is managed centrally by the domain controller, which holds a credential database containing hashed credentials for all domain accounts.

One workstation is used by a standard domain user and runs an email client application. The email client's inbox contains an email in which credentials for a second domain user account are stored in plaintext, making the email store a credential source in its own right rather than a store of harvested or dumped credentials. The second user account has access to the second workstation.

The domain uses Kerberos authentication. When a user on the second workstation requests access to a domain service, the domain controller issues a service ticket encrypted with the target account's credential hash. A user with access to the second workstation can request such a ticket for the domain administrator account and extract the encrypted hash from it for offline cracking, providing a path to domain administrator credentials without requiring direct access to the domain controller's credential database. The domain administrator account holds elevated access to both workstations and the domain controller.

There are no software vulnerabilities in this environment and no in-memory credential stores on the workstations. All credential acquisition paths run through the email store, the Kerberos ticket mechanism, or the domain controller's central credential database. The central credential database is therefore the ultimate high-value target, as compromising it yields credentials for every account in the domain.
